# [完全手搓一个博客](https://github.com/myogg/Gitblog/issues/142)

### 一个纯粹的手工作坊：我的博客成长记

在这个追求“开箱即用”和“一键部署”的时代，我却固执地选择了一条更“笨”的路——亲手搭建一个纯粹手工搓出来的博客。

它没有华丽的商业模板，也没有复杂的数据库支撑。它的核心逻辑非常简单甚至有些“硬核”：通过读取 GitHub 上的 Issue 内容，来动态生成我的博客文章。

#### 1. 把“缺陷”变成“文章”

对于大多数开发者来说，GitHub Issue 是用来追踪 Bug 和管理项目的。但在我看来，它也是一个绝佳的“内容管理系统”。

每当我有一个新的想法，或者踩到了一个新的技术坑，我就会去新建一个 Issue。在这个熟悉的 Markdown 编辑器里，我可以尽情地写下代码片段、贴上截图、梳理逻辑。而我的博客，就像是一个勤劳的“搬运工”，通过 API 定期去读取这些 Issue，将它们转化为网页上的一篇篇日志。

这种方式，让我的写作成本降到了最低。我不需要维护复杂的本地文件结构，也不需要学习新的后台操作。我只需要像平时提交代码一样，去维护我的 Issue。甚至，Issue 原生的评论功能，直接就成了我博客的评论区。这种将开发工具与写作工具完美融合的体验，是任何现成博客程序都无法给予的。

#### 2. 代码即文章，文章即代码

这种“手工搓”的方式，让我对“内容”有了更深的理解。

在传统的博客里，文章是存放在数据库里的字符串；而在我的博客里，文章是 GitHub 上的一个个实体，它们拥有完整的版本控制历史。每一次修改，都是一次清晰的 Commit。这不仅仅是记录文字，更像是在记录一种“思考的轨迹”。

通过编写脚本去抓取和解析这些 Issue，我深入理解了前后端交互的细节，学会了如何优雅地处理 Markdown 渲染，也体会到了自动化构建带来的便利。这个博客本身，就是我技术学习的练兵场。

#### 3. 慢慢学习，静静沉淀

做这样一个博客，没有捷径可走。从最初只能显示一行字，到后来能自动分页、生成目录、支持搜索，每一步都是在不断的报错和调试中走过来的。

“慢慢学习”这四个字，就是这个博客的灵魂。我不急于求成，也不追求功能的大而全。今天学会了一个新的 CSS 技巧，就给博客加个暗黑模式；明天理解了服务端渲染，就优化一下首屏加载速度。它不是完美的，甚至带着一些稚气，但它真实地记录了我技术成长的每一个脚印。

在这个喧嚣的互联网上，我的博客或许只是一个不起眼的“手工作坊”。没有流量，没有广告，只有我一个人在这里，对着代码，对着文字，安静地打磨时光。

如果你也喜欢这种纯粹的感觉，欢迎常来坐坐。这里没有高深的理论，只有最朴实的记录，和一个开发者慢慢学习、不断前行的心路历程。
